Continuing the GalaPo project. Build backend APIs for listing management.

TECH: Next.js 16 Route Handlers, Supabase, TypeScript

REQUIREMENTS:

1. /app/api/business/listings/route.ts:
   - GET: (already built in Module 8, fetch user's listings)
   - POST: Create new listing
     - Requires authentication (business_owner role)
     - Body: {
         category_id, subcategory_id,
         business_name, short_description, full_description,
         tags, phone, phone_secondary, email, website,
         social_links, payment_methods,
         address, barangay_id, lat, lng, operating_hours,
         dynamic_fields: [{ field_id, value }],
         is_draft: boolean
       }
     - Validation:
       - business_name required, max 100 chars
       - short_description required, max 160 chars
       - phone required, PH format
       - category_id required, must exist
       - barangay_id required, must exist
       - lat/lng must be valid coordinates within Olongapo area
       - Validate dynamic fields: check required fields for the selected category
     - Process:
       1. Generate slug from business_name (ensure unique, append -2, -3 if duplicate)
       2. Set city_id to Olongapo
       3. Set owner_id to current user
       4. Set status to 'pending' (or 'draft' if is_draft)
       5. Insert listing
       6. Insert dynamic field values into listing_field_values
       7. Create notification for super admin: "New listing submitted: [name]"
       8. Return created listing with ID

2. /app/api/business/listings/[id]/route.ts:
   - GET: Fetch single listing by ID (owned by current user)
     - Include all related data for editing (images, dynamic fields, deals, events)
   - PUT: Update listing
     - Only owner can update their own listing
     - Body: same as POST
     - Critical field detection:
       - If business_name, category_id, subcategory_id, address, or barangay_id changed AND listing was 'approved':
         - Set status back to 'pending'
         - Notify admin: "Listing [name] updated and needs re-approval"
       - If only non-critical fields changed:
         - Keep status as 'approved'
     - Update listing_field_values (upsert: update if exists, insert if new, delete if removed)
     - Regenerate slug if business_name changed
   - DELETE: Soft delete (set is_active = false)
     - Only owner can delete
     - Doesn't actually delete — deactivates
     - Active subscriptions are NOT refunded (manual process)

3. /app/api/business/listings/[id]/images/route.ts:
   - GET: Fetch all images for a listing (owned by current user)
   - POST: Upload new image(s)
     - Accept multipart form data (multiple files)
     - Upload to Supabase Storage "listings" bucket (path: [listing_id]/[filename])
     - Create listing_images records
     - Set first uploaded image as is_primary if no primary exists
     - Max 10 images per listing
     - Validate: only images (jpg, png, webp), max 5MB each
   - PUT: Reorder images
     - Body: [{ id, sort_order }]
     - Update sort_order for each image
   - DELETE: Delete image
     - Query param: ?image_id=xxx
     - Delete from Storage AND database
     - If deleted image was primary, set next image as primary

4. /app/api/business/listings/[id]/logo/route.ts:
   - POST: Upload logo
     - Upload to Supabase Storage "logos" bucket (path: listings/[listing_id])
     - Update listing.logo_url
     - Delete old logo if exists
     - Validate: image only, max 2MB

5. /app/api/business/listings/[id]/draft/route.ts:
   - POST: Save listing as draft
     - Saves all current data without full validation
     - Status = 'draft' (add to enum if not exists)
     - Allows partial data (not all required fields needed)

6. /app/api/category-fields/[categoryId]/route.ts:
   - GET: Fetch dynamic fields for a given category
   - Query param: ?subcategory_id=xxx (optional, to include subcategory-specific fields)
   - Returns: ordered list of fields with type, options, validation rules
   - Public endpoint (no auth needed — used in both business form and public display)

7. /app/api/claim/route.ts:
   - POST: Submit claim request
   - Body: { listing_id, phone, notes }
   - File: claim proof document (uploaded to "claims" bucket)
   - Process:
     1. Verify listing exists and has no owner (is_pre_populated = true, owner_id = null)
     2. Verify user doesn't already have a pending claim for this listing
     3. Upload proof document
     4. Set listing claim_proof_url
     5. Set listing status to 'claimed_pending'
     6. Create notification for super admin: "New claim request for [listing name]"
     7. Return success

8. /app/api/claim/[listingId]/status/route.ts:
   - GET: Check claim status for a listing
   - Returns: { is_claimed, is_claim_pending, is_owner }
   - Public endpoint (used on listing detail page to show/hide claim button)

9. CREATE /lib/listing-helpers.ts:
   - generateUniqueSlug(businessName, existingSlugs): string
   - validateListingData(data, categoryFields): { isValid, errors }
   - detectCriticalChanges(oldListing, newData): boolean
   - formatOperatingHours(hours): validated JSONB
   - validateCoordinates(lat, lng): boolean (within Olongapo bounds)
   - OLONGAPO_BOUNDS = { north: 14.88, south: 14.78, east: 120.32, west: 120.23 }

Output all files with complete code. Include SQL for any schema additions (draft status enum value).
