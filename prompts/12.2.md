Continuing the GalaPo project. Build backend APIs for admin content management.

TECH: Next.js 16 Route Handlers, Supabase, TypeScript

REQUIREMENTS:

1. /app/api/admin/categories/route.ts:
   - GET: Fetch all categories as tree structure
     - Include: subcategories nested, listing count per category, field count
     - Include inactive categories (admin needs to see everything)
   - POST: Create new category
     - Body: { name, slug, description, icon, parent_id, sort_order, is_active }
     - Auto-generate slug if not provided
     - Validate: unique slug, valid parent_id if provided
     - Return created category

2. /app/api/admin/categories/[id]/route.ts:
   - GET: Single category with all fields and subcategories
   - PUT: Update category
     - Body: partial update of any field
     - If slug changed, validate uniqueness
     - If category has listings and is being deactivated, show warning count
   - DELETE: Delete category
     - Only if no listings reference this category
     - Also deletes associated category_fields
     - If listings exist, return 409 with listing count

3. /app/api/admin/categories/reorder/route.ts:
   - PUT: Reorder categories
   - Body: [{ id, sort_order, parent_id }]
   - Bulk update sort_order and optionally parent_id (for drag between groups)

4. /app/api/admin/category-fields/route.ts:
   - POST: Create new dynamic field
   - Body: { category_id, subcategory_id, field_name, field_label, field_type, is_required, placeholder, help_text, options, validation_rules, sort_order, is_active }
   - Validate: unique field_name within category+subcategory

5. /app/api/admin/category-fields/[id]/route.ts:
   - GET: Single field
   - PUT: Update field
   - DELETE: Delete field
     - Warning if listings have values for this field
     - Also deletes listing_field_values for this field

6. /app/api/admin/category-fields/reorder/route.ts:
   - PUT: Reorder fields within a category
   - Body: [{ id, sort_order }]

7. /app/api/admin/barangays/route.ts:
   - GET: All barangays with listing count (admin sees inactive too)
   - POST: Create new barangay
     - Body: { name, slug, sort_order, is_active }
     - Validate: unique slug
   
8. /app/api/admin/barangays/[id]/route.ts:
   - PUT: Update barangay
   - DELETE: Delete barangay (only if no listings reference it)

9. /app/api/admin/barangays/reorder/route.ts:
   - PUT: Bulk reorder
   - Body: [{ id, sort_order }]

10. /app/api/admin/pages/route.ts:
    - GET: All static pages
    - POST: Create new page
      - Body: { title, slug, content, meta_title, meta_description, is_published, sort_order }

11. /app/api/admin/pages/[id]/route.ts:
    - GET: Single page
    - PUT: Update page (content, title, SEO fields, published status)
    - DELETE: Delete page (with confirmation)

12. /app/api/admin/settings/route.ts:
    - GET: All site settings
    - PUT: Update settings
      - Body: { [key]: value } — partial update
      - Validate known keys
      - Store in site_settings table as key-value pairs
    - Specific validation:
      - Pricing values must be positive numbers
      - URLs must be valid format
      - AdSense IDs format check

13. /app/api/pages/[slug]/route.ts (PUBLIC):
    - GET: Fetch published static page by slug
    - Only returns published pages
    - Cache: revalidate every 30 minutes
    - Used by public-facing pages: /about, /contact, /terms, /privacy, /faq, /advertise

14. Create public static page routes:
    - /app/(public)/[pageSlug]/page.tsx — catch-all for static pages
    - Renders content from database
    - SEO: uses meta_title and meta_description from page
    - Styled consistently with public layout
    - Handles 404 for non-existent slugs
